# Data dictionary — SQL Injection Lab

Everything here is **synthetic — generated by Square 1 AI, free for learners.** No real users, credentials, customer data, or PII. This is a **self-contained, offline lab**: nothing connects to the internet, and every technique targets **only** the bundled local app you run on `127.0.0.1`.

> ⚠️ **Authorized use only.** The injection techniques in this kit are for testing the **provided** vulnerable app and the payload corpus that ship here. Do not point them at any system you do not own or have written permission to test.

## `app.db` — synthetic SQLite database

The database behind the bundled vulnerable Flask app. Rebuildable from `seed.sql`.

### Table `users`
| Column | Type | Description |
|---|---|---|
| `id` | int | Primary key. |
| `username` | string | Login name (e.g. `ava.kim`). |
| `password` | string | **Plaintext on purpose** — this is the lab's deliberately bad app. Never do this in production. |
| `role` | string | `user` or `admin`. |
| `secret` | string | Only the single `admin` row carries the planted **flag**. It is an **obviously-fake** string (`FLAG_sq1_EXAMPLE_..._REPLACE`) — not a real secret. Recovering it via injection proves the vulnerability; the fix must make it unreachable. |

### Table `products`
| Column | Type | Description |
|---|---|---|
| `id` | int | Primary key. |
| `name` | string | Product name (e.g. `Gadget Model 412`). |
| `category` | string | Widgets / Gadgets / Tools / Stationery / Hardware. |
| `price` | float | Price (AUD). |
| `listed` | int | `1` = public (returned by a normal search); `0` = **internal-only**, must never surface to an unauthenticated search. The vulnerable search **leaks** these under injection; a correct parameterized search never does. |

## `seed.sql`
A human-readable SQL dump (schema + rows) equivalent to `app.db`. Use it to inspect or rebuild the database.

## `payloads.csv` — labelled injection corpus

For training and evaluating your detector. One row per input string.

| Column | Type | Description |
|---|---|---|
| `payload` | string | The input string. |
| `label` | string | `benign` or `malicious`. |
| `technique` | string | For malicious rows: `union` / `boolean` / `error` / `auth-bypass`. Empty for benign rows. |
| `encoding` | string | `plain` or `url` (URL-encoded variant of a malicious payload). |

### Dirt / edge-cases you must handle
- **Benign inputs that legitimately contain quotes** — names like `O'Brien`, `d'Angelo`, and a product search `5'9" shelf`. A detector that flags every quote will have terrible precision. Distinguish a quote that *breaks out of a string literal* from one that is ordinary data.
- **URL-encoded malicious payloads** (`encoding = url`) — the same attack, percent-encoded (e.g. `%27%20OR%201%3D1`). A naive substring scan for `UNION`/`OR` misses these. Decode before you classify.
- **Error / boolean-blind payloads leak via side-channels, not rows** — error-based and boolean payloads may return zero rows yet still reveal information through error messages or response differences. Your write-up should note that "rows returned" is only one of several success signals.
